================================================================================
                    TOKEN USAGE IN MCP TOOL CALLING - EXPLAINED
================================================================================

PART 1: WHY ALL TOOL DEFINITIONS ARE SENT ON EVERY MESSAGE
================================================================================

When you use MCP (Model Context Protocol) with an LLM, here's what happens:

STEP 1: User sends a message
STEP 2: The system sends to LLM:
        - ALL available tool definitions (so LLM knows what's available)
        - User's message
        - Conversation history
STEP 3: LLM reads everything and decides which tool to use
STEP 4: LLM outputs the tool call

KEY POINT: The LLM cannot choose a tool it doesn't know about.
           It MUST see ALL tool definitions to make a selection.


EXAMPLE WITH 227 SPECIFIC TOOLS:
--------------------------------

User asks: "Get all users"

What LLM receives in context:
┌─────────────────────────────────────────────────────────────────┐
│ TOOL DEFINITIONS (all 227 sent every time):                     │
│                                                                 │
│ Tool 1: qradar_get_offenses                                     │
│   - description: "Get SIEM offenses"                            │
│   - params: {status, severity, filter}                          │
│                                                                 │
│ Tool 2: qradar_get_users          ← LLM will pick this one     │
│   - description: "Get QRadar users"                             │
│   - params: {filter, fields}                                    │
│                                                                 │
│ Tool 3: qradar_get_rules                                        │
│   - description: "Get correlation rules"                        │
│   - params: {rule_type, filter}                                 │
│                                                                 │
│ ... (224 more tools) ...                                        │
│                                                                 │
│ Tool 227: qradar_delete_reference_set                           │
│   - description: "Delete a reference set"                       │
│   - params: {name}                                              │
├─────────────────────────────────────────────────────────────────┤
│ USER MESSAGE: "Get all users"                                   │
└─────────────────────────────────────────────────────────────────┘

LLM Output: {"tool": "qradar_get_users", "params": {}}

TOKEN COUNT: ~22,000 tokens (for 227 tool definitions) + user message

4-TOOL APPROACH USES 22x FEWER TOKENS!

PART 2: HOW LLM IDENTIFIES CORRECT ENDPOINT USING qradar_get TOOL
================================================================================

The LLM uses its training knowledge + tool description to construct the endpoint.

METHOD 1: LLM KNOWS COMMON ENDPOINTS (Direct Call)
--------------------------------------------------

User: "Get all users"

LLM thinks:
  - User wants users
  - I have qradar_get tool
  - I know QRadar API uses /config/access/users for users
  - I'll use that endpoint

LLM Output: qradar_get(endpoint="/config/access/users")

This works because:
  - Claude/GPT are trained on QRadar API documentation
  - Common endpoints like users, offenses, rules are well-known
  - LLM can infer the endpoint from its training


METHOD 2: USER DOESN'T KNOW ENDPOINT → USE qradar_discover FIRST
----------------------------------------------------------------

User: "Create a reference set for blocked IPs"

LLM thinks:
  - User wants to create something
  - I don't know the exact endpoint for reference sets
  - I should discover it first

STEP 1 - LLM calls discover:
  qradar_discover(search="reference_data/sets", method="POST")

STEP 1 - Tool returns:
  {
    "endpoints": [
      {
        "path": "/reference_data/sets",
        "method": "POST",
        "description": "Create a new reference set",
        "parameters": [
          {"name": "name", "required": true, "type": "string"},
          {"name": "element_type", "required": true, "type": "string"}
        ]
      }
    ]
  }

STEP 2 - LLM now knows exact endpoint and calls execute:
  qradar_execute(
    method="POST",
    endpoint="/reference_data/sets",
    params={"name": "blocked_ips", "element_type": "IP"}
  )


METHOD 3: LLM USES EXAMPLES FROM TOOL DESCRIPTION
-------------------------------------------------

PURPOSE OF EXAMPLES IN TOOL DESCRIPTION:
The examples serve as a "cheat sheet" for the LLM. Instead of relying only on 
its training knowledge (which might be outdated or incomplete), the LLM can 
directly copy the endpoint pattern from the examples.

This reduces errors and hallucinations because:
  - LLM doesn't have to guess the endpoint format
  - Exact syntax is provided (e.g., "/siem/offenses" not "/api/siem/offense")
  - Common use cases are pre-documented

Your 4 tools and their purposes:

  ┌────────────────────────────────────────────────────────────────────┐
  │ Tool 1: qradar_get                                                 │
  │   Purpose: GET requests - Fetch/retrieve data from QRadar          │
  │   Use for: Listing resources, getting specific items, querying     │
  │                                                                    │
  │ Tool 2: qradar_delete                                              │
  │   Purpose: DELETE requests - Remove resources from QRadar          │
  │   Use for: Deleting reference set entries, saved searches, etc.    │
  │                                                                    │
  │ Tool 3: qradar_discover                                            │
  │   Purpose: Find endpoints and their schemas                        │
  │   Use for: When LLM doesn't know exact endpoint path/parameters    │
  │                                                                    │
  │ Tool 4: qradar_execute                                             │
  │   Purpose: POST/PUT/PATCH requests - Create or modify resources    │
  │   Use for: Creating items, updating data, running AQL searches     │
  └────────────────────────────────────────────────────────────────────┘

Example of qradar_get tool with examples in description:

  Tool: qradar_get
  Description: "Fetch data from QRadar. Use for listing or retrieving resources.
  
  Examples:
  - List offenses: endpoint="/siem/offenses"
  - Get offense: endpoint="/siem/offenses/123"
  - Filter: endpoint="/siem/offenses", filter="status=OPEN"
  - List users: endpoint="/config/access/users"
  - System info: endpoint="/system/about"
  "

HOW IT WORKS:

When user asks "Get all users", LLM sees the example:
  "List users: endpoint="/config/access/users""

And uses exactly that endpoint! No guessing needed.

When user asks "Show system information", LLM sees:
  "System info: endpoint="/system/about""

And uses exactly that endpoint!

WHY THIS IS POWERFUL:
  - Examples are sent WITH the tool definition (part of the ~1,000 tokens)
  - No extra token cost for guidance
  - LLM learns correct patterns from examples
  - Reduces hallucination of wrong endpoints

PART 3: WHEN LLM MIGHT NOT KNOW THE ENDPOINT
================================================================================

For obscure or complex endpoints, LLM might not know them from training.

Example: "Get the QNI host configurations"

LLM might not know the exact path. In this case:

OPTION A: LLM guesses (risky, might be wrong)
  qradar_get(endpoint="/qni/hosts")  ← might be wrong!

OPTION B: LLM uses discover first (safer)
  qradar_discover(search="qni host")
  
  Returns: {path: "/qni/hosts/{host_id}/configs", method: "GET"}
  
  Then: qradar_get(endpoint="/qni/hosts/1/configs")


This is why you have qradar_discover tool - it helps LLM find endpoints
it doesn't know from training.


================================================================================
SUMMARY
================================================================================

1. ALL tool definitions are sent on EVERY message (LLM needs to see options)

2. 227 specific tools = ~22,000 tokens per message
   4 generic tools = ~1,000 tokens per message
   
3. The 4-tool approach is 22x more token efficient

4. LLM identifies endpoints using:
   - Training knowledge (knows common QRadar endpoints)
   - Examples in tool description
   - qradar_discover tool for unknown endpoints

5. For large APIs like QRadar (728 endpoints), the 4-tool approach is more 
   efficient than creating hundreds of specific tools.

THE 4-TOOL APPROACH IS CORRECT AND OPTIMAL FOR QRADAR.
